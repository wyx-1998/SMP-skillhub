# 隐患辅助评估

按“发现实体 → 读取登记记录与字段约束 → 组织评估输入 → 执行方法论 → 转换并校验保存数据 → 输出结论 → 用户确认后保存”完成整个流程。

## 一、定位实体和记录

1. 遵守 系统提示词。如果当前页面或上下文已经提供唯一 `entityCode` 和记录主键，直接进入实体描述；否则按以下顺序发现：
   - 调用 `discoverModules()` 获取模块与实体菜单；
   - 选择目标模块后调用 `listEntitySummaries(module)` 获取实体索引；
   - 根据隐患业务语义确定唯一实体。
2. 调用 `describeEntity(entityCode)` 读取字段、类型、字典、必填规则、子实体和引用关系。不得用界面截图、历史字段名或经验替代实体描述。
3. 调用 `searchRecords(entityCode, where, pageNum, pageSize)` 读取当前登记记录。`where` 必须使用当前记录的唯一条件，不得扩大查询范围。
4. 实体、记录主键或唯一查询条件无法确定时停止，返回技术阻断原因，不猜测实体或记录。
5. `describeWorkflow` 当前不返回流程定义，不得依赖它推断节点、状态或允许动作。

## 二、组织评估输入

1. 从当前唯一记录中收集登记描述、图片、附件、地点、对象、部门、登记时间和已有检测或状态数据。
2. 保留每项数据的来源、原值和对应记录，组成只读登记证据包；方法论执行完成前不得修改登记数据。
3. 从 `describeEntity` 返回的字段中提取必填字段、条件必填字段和承载核心评估结论的字段，结合字段类型、字典选项和条件规则形成评估字段约束包。每项约束包含稳定语义标识、字段含义、是否必填、选择方式、可读选项、默认值和适用条件；实体未返回的属性标记为未提供，不自行补充。字段含义不能确定时停止，不猜测语义标识。
4. 将部门、人员、具体日期、记录关系等管理性字段保留在约束包中并标记为业务补充项，不用这些字段判断隐患是否成立、隐患因素或隐患等级。
5. 同时保留内部字段绑定，包括“语义标识 → 字段代码”和“可读选项 → 字典保存值”。字段代码和保存值只用于后续转换，不作为隐患事实或评估依据。
6. 将登记证据包和评估字段约束包共同组成方法论输入。业务默认值只表示界面初始状态，不视为登记事实或预设评估结论。

## 三、形成评估结论

1. 完整读取并执行[六步隐患评估方法论](references/rht-hidden-trouble-methodology.md)，同时输入登记证据包和评估字段约束包。
2. 允许方法论按其内部指引加载所需知识资源。
3. 接收方法论形成的“标准评估结果结构”，包括评估结论、隐患事实、分类分级、判断依据、整改要求、验收条件、人工补充项和业务必填语义响应。
4. 在数据转换前保持语义中间结果不变，不重新判断或改写方法论结论。

## 四、转换并校验保存数据

1. 使用组织评估输入时保留的内部字段绑定，将业务必填语义响应转换为当前实体的字段代码和字典保存值，不重新搜索或推断字段语义。
2. 按响应状态处理每项约束：
   - **已确定**：转换可读结果并保留判断依据；
   - **待补充**：不生成字段值，写入人工补充项；该字段必填时阻止保存；
   - **不适用**：仅在实体条件规则允许时不填写；实体仍要求必填时标记业务规则冲突并阻止保存。
3. 不使用实体默认值覆盖方法论响应。只有部门、人员、具体日期等不参与专业判断的管理性字段，才能依据登记数据、实体关系和系统明确提供的默认规则补充；没有依据时不得凭空指定。
4. 方法论给出的整改时限要求只表达立即整改、使用前整改或限期整改等语义。具体整改日期仅在现有规则能够确定时转换；否则作为必填缺失项处理。
5. 根据实体字段类型转换日期、文本、字典、引用和其他保存格式。展示值使用可读文本，内部保存计划使用实体允许的值。
6. 完成必填、条件必填、字典、字段类型、记录主键和关联关系校验。系统没有字段承载的补充说明放入摘要，不扩展实体结构。
7. 转换阶段不得重新选择评估结果、隐患分类、隐患类型、可能后果等级、隐患级别、整改计划或隐患因素，不得改变隐患事实和判断依据。
8. 法规依据只能使用方法论输出的真实、可追溯名称、版本、条款位置和原文。

## 五、输出结论

基于评估结果使用可读性高的 Markdown 输出：

- 先给出评估结果和一句话摘要；
- 按评估字段约束逐项展示字段含义、可读结果、响应状态和判断依据；
- 分别展示隐患事实、可能后果、隐患等级、整改要求、整改时限要求和验收条件；
- 集中列出待补充项、不适用项和不能保存的原因；
- 不展示内部字段代码、字典保存值、记录主键或关联数据。

## 六、确认和保存

1. 先输出评估结论，等待用户明确确认。
2. 用户确认后，重新调用 `describeEntity(entityCode)` 和 `searchRecords`，确认实体定义、评估字段约束、内部字段绑定、记录对象和当前数据未发生影响结论或保存计划的变化。
3. 仅更新评估字段时，调用 `saveRecords(entityCode, data)`，携带当前记录主键和内部保存值；不得把展示文本当作字典保存值。
4. 只有当前上下文明示需要执行独立业务动作并提供准确 `actionName` 时，才调用 `performAction(entityCode, id, actionName, args)`；不得自行推断动作名或动作合法性。
5. MCP 返回结构化错误时，根据 `type`、`code`、`retryable` 和 `details` 处理。参数或约束错误修正后重试；认证、权限、动作不允许、记录不存在或不可重试错误直接说明，不伪装成保存成功。
6. 证据、方法论结论、记录对象、内部字段绑定或保存计划发生变化时停止保存，重新执行受影响阶段并再次取得用户确认。

## 执行门禁

- 不绕过 MCP 工具直接拼接业务 HTTP 请求。
- 没有同时向方法论提供登记证据包和评估字段约束包，不输出评估字段。
- 不把实体必填、默认值或字典选项改写成登记事实或预设结论。
- 不猜测字段含义、条件规则或可读选项；评估字段约束无法确定时停止输出可保存数据。
- 不把图片候选线索、可能后果或人工补充内容改写成已确认的隐患事实。
- 不编造实体、记录、现场事实、字段、字典值、责任主体、期限、动作名或法规依据。
- 存在必填项待补充、必填项不适用冲突或保存数据校验失败时，不进入确认保存。
- 没有用户确认，不保存或执行动作。
