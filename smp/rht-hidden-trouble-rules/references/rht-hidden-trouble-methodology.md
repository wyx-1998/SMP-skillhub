# 六步隐患评估方法论

本方法用于评估已经登记的隐患。它基于登记时已有的描述、图片、附件和数据形成明确结论，并按照运行时提供的评估字段约束完整响应业务所需语义。本方法不直接访问业务系统，也不内置具体字段代码、字典保存值或实体结构。

2. 隐患定义

9. 质量检查

## 一、方法定位

隐患评估的输入是一条已经登记的隐患记录。评估不重新开展全场景安全检查，只判断当前登记内容是否构成隐患、能否进入整改，并形成后续处置所需的明确结论。

### 方法输入

本方法接收登记证据包和评估字段约束包。

登记证据包可包含：

- 登记描述；
- 图片、视频和附件；
- 地点、对象、部门、登记时间等上下文；
- 检测数据、设备状态和其他原始记录。

评估仅使用登记时已经取得的材料。登记人与评估人分离时，不假设能够返回现场补录信息；材料缺失、冲突或不完整的内容按人工补充项处理。

评估字段约束包由应用层根据当前实体中的必填字段、条件必填字段和承载核心评估结论的字段生成，每项可包含：

- 稳定语义标识和字段含义；
- 是否必填及条件必填规则；
- 字段类型和选择方式；
- 当前允许的可读选项；
- 界面默认值；
- 适用条件和管理性补充标记。

字段代码、字典保存值、记录主键和关联关系不属于方法论输入。

评估字段约束只规定需要回答的业务语义和允许的表达范围，不属于登记证据，不能证明隐患事实。部门、人员、具体日期等管理性字段不参与隐患成立、隐患因素和隐患等级判断。

核心链路为：

```text
登记证据包 + 评估字段约束包（输入）
→ 1. 提取并标准化登记主张
→ 2. 核对登记主张与证据
→ 3. 对照安全基准并判断隐患是否成立
→ 4. 评估隐患影响与处置属性
→ 5. 匹配重大隐患判定规则
→ 6. 形成结论、整改验收要求和业务语义响应
```

评估主结论只有两种：

- **评估通过**：登记内容能够识别或明确指向人的不安全行为、物的不安全状态或管理缺陷；信息不完整、证据冲突、分级要件不足或整改信息缺失不改变通过结论，统一列为人工补充项；
- **评估未通过**：依据现有客观事实能够确认登记内容不符合隐患定义，三类隐患事实均不存在。

“评估通过”表示该登记可以进入整改，不表示现场状态安全，也不表示整改已经完成。

## 二、隐患定义

隐患是指生产经营单位违反安全生产法律、法规、规章、标准、规程和安全生产管理制度的规定，或者因其他因素在生产经营活动中存在可能导致事故发生的物的危险状态、人的不安全行为和管理上的缺陷。

隐患有三类基本表现，任意一类成立即可构成隐患，不要求三类同时出现：

- **人的不安全行为**：人员资格、授权、操作方式或作业行为不符合安全要求；
- **物的不安全状态**：设备、设施、工具、材料或环境处于危险、失效、防护不足或参数异常状态；
- **管理缺陷**：制度、责任、流程、培训、检查、许可、监督或资源保障存在缺失、失效或未落实。

判断隐患是否成立有两条路径，满足其中一条即可：

1. **规范偏离路径**：有明确适用的安全要求，登记事实违反或未达到该要求；
2. **潜在致害路径**：即使没有取得具体条款，登记事实属于上述三类表现之一，并存在合理、可信、可解释的潜在致害路径。

隐患回答的是：

> 当前存在什么不安全行为、不安全状态或管理缺陷？

例如：

- 未取得相应资格的人员从事依法需要持证的电工作业；
- 配电箱箱门破损，内部带电部件可被接触；
- 有限空间内氧含量低于允许范围；
- 强制通风设备无法启动；
- 防护栏杆缺失；
- 联锁报警装置失效；
- 关键安全检查制度缺少必要的执行和记录要求。

完整隐患事实通常包括：

```text
人员、对象或管理环节 + 地点或范围 + 不安全行为、不安全状态或管理缺陷 + 登记时点 + 证据来源
```

可能后果可以单独分析，但不属于隐患事实，不能代替三类隐患事实中的任何一类。

## 三、基本原则

1. **围绕登记隐患评估**：不扩展为整个现场或作业的全面检查。
2. **登记描述是主要观察记录**：登记人对人员行为、物的状态或管理缺陷的明确描述属于主要证据，不要求图片重新证明全部内容。
3. **图片用于一致性校验**：图片能够支持或反驳登记描述，也能提供补充线索，但不能证明未拍摄区域和内部功能。
4. **主结论必须明确**：输出评估通过或评估未通过，不把“待复核”作为第三种主结论。
5. **信息缺失不否定隐患**：缺失信息统一形成可执行的人工补充项，不得用评估未通过代替补充流程。
6. **安全基准决定规范偏离是否成立**：不能把“没有采取某措施”自动等同于隐患，应确认当前人员、对象或管理环节确实适用该要求。
7. **允许有边界的专业判断**：可以根据登记事实和通常工况判断分类、影响和处置，但必须说明依据，不得补造现场参数。
8. **隐患程度与重大隐患分开**：后果严重或问题明显，不等于自动命中重大隐患标准。
9. **整改状态与发现时点分开**：当场整改不抹去发现时点已经成立的隐患事实。
10. **证据决定判断，字段约束决定表达**：隐患成立、事实和等级只能根据登记证据与适用安全基准判断，不能根据必填字段、可选项或默认值反推。
11. **完整响应评估约束**：对每个输入的评估字段约束给出已确定、待补充或不适用状态，避免把专业判断留到数据转换阶段。
12. **默认值不是评估结论**：界面默认选择必须经过方法论重新判断，不得直接继承。

## 四、六步评估模型

### 第一步：提取并标准化登记主张

#### 1. 拆分登记内容

把登记描述拆成三类内容：

- **核心观察**：人员、对象或管理环节，地点，不安全行为、不安全状态或管理缺陷，当前状态，以及能够落到现场的受摩擦、挤压、碰撞、积聚等暴露条件；
- **后果判断**：由隐患事实和暴露条件进一步引出的短路、漏电、失控、伤害、损失等结果；
- **整改建议**：“应当修复、增加、更换”等动作。

只用核心观察形成标准化隐患描述。后果判断和整改建议分别进入后续步骤。

#### 2. 形成描述草案

```text
【地点、作业、对象或管理环节】存在【不安全行为、不安全状态或管理缺陷】。
```

登记材料明确给出运行、停用、隔离或其他当前状态时，再补充状态；没有给出时不写“未知状态”。

#### 3. 建立证据索引

记录每项核心观察在登记证据包中的来源和原值。此处只建立对应关系，不判断证据支持、冲突或缺失状态。

第一步输出登记主张、标准化描述草案和证据索引。

### 第二步：核对登记主张与证据

#### 1. 核心事实状态

将登记主张拆成原子事实，每项标记：

- **支持**：登记描述明确，且图片、附件或上下文未出现直接冲突；
- **增强支持**：登记描述明确，并有图片、检测数据或附件进一步印证；
- **冲突**：图片、附件或其他直接证据否定登记描述；
- **补充信息缺失**：核心隐患事实能够识别，但数量、参数、使用状态或范围等次要信息没有提供。

登记描述清楚且没有反证时，可以支持核心事实，不要求每项都由图片独立复现。模型识别结果只能作为候选线索，不能单独形成核心事实。

#### 2. 核对材料对应关系

核对登记描述、图片、附件和上下文是否对应：

- **对象**：是否指向同一人员、设备、设施或管理环节；
- **地点**：是否属于同一作业地点、设备位置或管理范围；
- **时间**：是否反映同一登记时点；整改后、停用后或其他时点的材料不能直接否定发现时的隐患事实；
- **事实**：图片、检测数据和附件是支持、增强支持，还是直接反驳登记主张；
- **来源**：每项核心观察是否能够对应到具体登记材料及其原值。

材料不属于同一对象、地点或时间，或者材料之间存在冲突时，如实记录差异，不在本步骤作出评估通过或评估未通过结论。

第二步输出事实证据表、材料对应关系、证据冲突和证据类人工补充项。

### 第三步：对照安全基准并判断隐患是否成立

#### 1. 确定安全基准

安全基准说明相关人员、对象或管理环节本应达到的安全状态：

```text
人员、对象或管理环节 + 应有状态或禁止行为 + 适用条件 + 判断方法 + 依据来源
```

来源优先级通常为：

1. 现行法律法规、强制性标准和监管规定；
2. 设计文件、设备说明、检测阈值和经批准的技术方案；
3. 企业技术标准、检维修标准和检查标准；
4. 可靠的专业资料和经确认的专业判断。

没有取得准确条款，不必然阻断一般隐患评估。可以使用可靠、可解释的工程、作业或管理安全基准，但不得编造法规名称和条款位置。

#### 2. 判断成立路径

优先检查规范偏离路径：确认安全要求是否适用，再把登记事实与安全基准比较。没有取得具体条款时，检查潜在致害路径：确认三类隐患事实中的具体表现、可信触发条件和可能影响。

- 存在人的不安全行为、物的不安全状态或管理缺陷，并满足任一成立路径：隐患成立，评估通过；
- 登记内容指向隐患事实，但证据、参数、范围、状态或安全要求适用条件不完整：评估通过，列出人工补充项；
- 登记主张与其他材料冲突，或者材料不属于同一对象、地点或时间：不直接否定登记，评估通过，列出人工核对项；
- 登记事实符合适用安全要求，且客观事实能够确认三类隐患事实均不存在，也不存在可信潜在致害路径：隐患不成立，评估未通过。

#### 3. 使用评估未通过的唯一口径

评估未通过只用于表达：

> 根据登记描述、图片、附件、适用安全要求和其他现有资料，可以明确确认该登记不存在人的不安全行为、物的不安全状态或管理缺陷，也不存在规范偏离或可信潜在致害路径，因此不属于隐患。

以下情形均不得直接判评估未通过：

- 只有可能后果，尚未写清当前隐患事实；
- 只写未采取某项措施，尚未说明适用条件；
- 人员、对象、管理环节或隐患类别描述不完整；
- 图片或附件与登记描述冲突；
- 登记材料可能不属于同一对象或同一时间；
- 缺少区分重大隐患与一般隐患的要件；
- 暂时无法形成完整整改方案。

这些情形统一评估通过，并将缺失、冲突或待确认内容列为人工补充项，由后续人员补充后进入整改或重新判级。

第三步输出评估结论、隐患类别、安全基准、登记事实、成立路径、判断依据和人工补充项。

### 第四步：评估隐患影响与处置属性

#### 1. 隐患程度

| 隐患程度 | 判断口径 |
|---|---|
| 关键安全控制失效 | 人员资格或授权不满足关键要求，物的安全功能失效，或者关键管理控制缺失、失效 |
| 安全控制能力下降 | 行为、状态或管理缺陷已经削弱控制能力，但关键控制尚未完全失效 |
| 一般偏差 | 存在不符合安全基准的行为、状态或管理问题，暂未证明关键控制失效 |

缺少内部测试或精确参数时，根据已经确认的事实选择能够成立的最低隐患程度，并把需要验证的资格、功能、参数或管理要求写入验收条件，不使用“程度未知”代替整个评估。

#### 2. 当前状态

区分不安全行为正在发生或已经停止、对象正在使用或已经停用、管理缺陷仍然存在或已经消除，以及是否已采取临时控制。登记材料没有提供当前状态时，不补造状态；根据已经确认的隐患事实给出恢复作业或使用前必须满足的条件。

#### 3. 可能后果

选择与隐患事实及其能量来源相符的“合理可预见最严重后果”，而不是最轻后果，也不是脱离事实的最坏想象。

分析链路为：

```text
不安全行为、不安全状态或管理缺陷 + 可信触发条件 + 影响路径 + 人员、设备、生产或环境后果
```

后果单独表达，不写进隐患事实。没有依据时不编造具体伤亡人数、损失金额或介质性质。

评估字段约束包含后果严重程度等单选语义时，按照合理可预见的最严重人员伤害后果，从当前允许的可读选项中选择一项，并单独说明选择依据。可能后果描述与后果严重程度选项必须分别输出；现有事实无法支持任何选项时标记待补充，不得使用默认值代替判断。

#### 4. 分类和整改属性

根据当前登记隐患的主导类别和直接表现，形成业务应用所需的隐患分类、隐患类型层级、隐患因素、影响和整改属性等语义结果。存在对应评估字段约束时，从当前允许的可读选项中选择，并说明事实依据；方法论不输出字段代码或字典保存值。

单选维度只选择当前隐患事实的主导项，不把所有可能原因同时列入。现有选项不能准确表达评估判断时标记待补充，并说明业务规则冲突，不选择语义错误的近似项。

#### 5. 处置要求

| 处置要求 | 适用情形 |
|---|---|
| 立即停止、隔离或停用 | 不安全行为正在发生，关键安全控制已经失效，或者对象正在使用并持续影响安全条件 |
| 作业或使用前整改 | 行为已经停止、对象已经停用或当前状态未提供，恢复前必须消除隐患 |
| 限期整改 | 一般偏差，现有控制能够维持到整改完成 |

第四步输出隐患程度、可能后果、后果严重程度、分类语义、类型层级、隐患因素和处置要求。

先读总览、适用范围和判定总则，再在已选目录内组合检索人员、资格、行为、对象、状态、管理、阈值和失效等关键词。知识库只用于路由、理解构成要件和辅助解释，不单独证明标准现行有效，也不替代正式原文。

#### 2. 确认适用性

依次确认：

- 行业和法域；
- 企业、场所或设备类型；
- 相关人员、对象或管理环节及对应隐患事实；
- 标准版本及有效时间；
- 是否存在优先适用的专门规定。

#### 3. 拆解构成要件

将候选规则拆成：

```text
适用对象 + 场景条件 + 客观状态 + 数量或程度阈值 + 时间状态 + 逻辑关系 + 排除条件
```

逐项标记满足、不满足或材料未提供，并绑定证据。必须保留规则中的“并且、或者、之一、达到、超过、失效、继续使用”等逻辑。

#### 4. 形成隐患等级

- **重大隐患**：适用性明确，全部必要要件满足，且不存在排除情形；
- **一般隐患**：没有适用的重大隐患候选规则，或者候选规则存在明确不满足的必要要件，而隐患仍然成立；
- **判级信息待补充**：存在可信的重大隐患候选规则，但必要要件尚未全部确认。不得因此判评估未通过；按照已确认事实形成当前等级建议，列明缺失要件，人工补充后重新判级。

语义相似、可能后果严重或经验判断不能补足重大隐患的缺失要件。未命中重大隐患标准不等于隐患不存在。

评估字段约束要求在一般隐患和重大隐患之间单选时，只有判级要件充分才能输出对应可读选项。处于判级信息待补充状态时不得使用默认等级填充，相关约束响应标记为待补充。

### 第六步：形成结论、整改验收要求和业务语义响应

#### 1. 主结论

- **评估通过**：人的不安全行为、物的不安全状态或管理缺陷至少一类成立或被登记事实明确指向；材料缺失不改变通过结论；
- **评估未通过**：客观事实能够确认三类隐患事实均不成立，不属于隐患定义。

评估未通过必须说明不符合隐患定义的客观依据，不能以“资料不足、证据冲突、无法分级、无法整改”等理由判未通过。

#### 2. 人工补充项

将所有信息缺失、证据冲突和后续判断要求转化为：

```text
补充对象 + 需要补充或核对的内容 + 推荐资料或核验方式 + 补充后影响的结论或整改事项
```

人工补充项用于完善分级、整改和验收，不改变评估通过的主结论。

#### 3. 整改要求

整改措施直接针对不安全行为、不安全状态或管理缺陷：

- 恢复、修复、更换或增设必要的安全设施；
- 立即停止不安全行为，核验人员资格、授权和作业条件；
- 将异常参数恢复到允许范围；
- 停用、隔离或限制有缺陷的对象；
- 补齐影响安全控制有效性的管理要求；
- 完成功能测试和状态验证。

#### 4. 验收条件

把需要在整改后确认的信息和整改效果验证转化为可执行条件：

```text
验收对象 + 应达到的状态 + 验证方式 + 合格标准
```

例如：

```text
验收通风设备的安全功能；采用现场启动测试和运行参数记录验证；设备能够稳定启动且参数达到规定范围方可通过。
```

#### 5. 是否需要提交整改计划

- 单点修复、更换、清理、固定或补充防护即可完成的，一般不需要提交专项整改计划；
- 涉及停产停用、技术改造、多部门协同、较长周期、重大隐患或复杂验证的，需要提交整改计划；
- 重大隐患必须选择需要提交整改计划，不得使用界面默认值覆盖。

存在对应评估字段约束时，从当前允许的可读选项中选择需要或不需要，并保留选择依据。

#### 6. 整改时限要求

根据第四步的处置要求输出立即整改、停止作业或使用前整改、限期整改等时限语义。方法论不凭空生成具体日期；责任主体和具体整改日期应依据组织关系、管理要求和现有上下文确定。

#### 7. 响应评估字段约束

对评估字段约束包中的每一项输出：

```text
语义标识 + 字段含义 + 可读结果 + 响应状态 + 判断依据或缺失信息
```

响应状态只有三种：

- **已确定**：登记证据和评估过程足以确定结果；选择字段必须使用当前允许的可读选项；
- **待补充**：现有材料不足以确定结果，列明需要补充的事实及其影响，不使用默认值填充；
- **不适用**：该字段在当前评估结论下没有有效业务含义，并说明原因。

评估未通过时，与隐患类型、后果、等级、因素、整改计划和整改要求等仅对已成立隐患有意义的约束应标记不适用，不得生成虚假默认结果。部门、人员、具体日期等管理性字段没有现成依据时标记待补充，不作专业推断。

方法论只输出语义标识和可读结果，不输出字段代码、字典保存值、记录主键或关联数据。

## 五、证据使用规则

### 1. 登记描述

登记描述是登记人的现场观察记录。只要相关人员、对象或管理环节及其不安全行为、不安全状态或管理缺陷表达明确，且没有更强证据反驳，就可以支持核心事实。

描述中的不安全行为、现场暴露条件和管理缺陷可以支持安全基准判断；由其进一步推导的可能后果、原因推测和整改建议需要拆开使用，不能全部当作已经确认的现场事实。

### 2. 图片和视频

图片和视频用于：

- 核对对象、地点和外观状态；
- 增强或反驳登记描述；
- 识别与当前登记隐患直接相关的补充事实。

静态图片不能证明人员是否取得资格、管理制度是否缺失、设备内部功能、未拍摄区域、持续时间和精确参数。图片识别模型输出只是候选线索。

### 3. 证据冲突

出现直接冲突时，优先采用时间更接近登记时点、对象定位更清楚、测量方式更可靠的证据。冲突暂时无法消解时，保留评估通过结论并形成人工补充项；只有更强客观证据能够确认三类隐患事实均不存在时，才评估未通过。

### 4. 信息缺失

信息缺失不改变评估通过结论。根据影响范围分别写入隐患事实补充、判级要件补充、整改信息补充或验收条件，不得把缺失信息转换成评估未通过。

## 六、隐患描述规范

根据隐患类别选择描述结构：

```text
人的不安全行为：【地点或作业】由【资格、授权或操作方式不符合安全要求的人员】实施【具体作业】。
物的不安全状态：【地点或对象】存在【危险、失效、防护不足或参数异常状态】。
管理缺陷：【管理环节】存在【制度、责任、流程、检查、许可、监督或资源保障缺陷】。
```

登记材料明确提供当前状态时，可以补充：

```text
【对应隐患事实】，当前处于【已确认状态】。
```

只允许写入：

- 相关人员、作业、对象或管理环节；
- 不安全行为、不安全状态或管理缺陷；
- 必要的地点、数量和范围；
- 证据明确提供的当前状态。

不得写入：

- “可能导致、容易造成”等后果判断；
- 整改措施、责任人和期限；
- 证据没有提供的参数、数量、介质和持续时间。

## 七、整改与验收

整改分为：

- **临时控制**：在永久整改前停用、隔离、限制使用或设置可靠防护；
- **永久整改**：修复、更换、消除异常或恢复设计安全功能；
- **验收验证**：用测试、记录或现场状态证明缺陷已经消除。

验收不只检查“是否做了整改动作”，还要确认：

1. 原不安全行为、不安全状态或管理缺陷是否消除；
2. 安全功能是否恢复；
3. 参数、结构和状态是否达到安全基准；
4. 是否引入新的缺陷；
5. 临时控制是否可以解除。

已经当场整改不改变发现时点的隐患定性，应保留评估通过结论，并把当前整改状态和验收结果另行记录。

## 八、标准评估结果结构（语义中间层）

本结构是六步方法论形成的中间语义结果，不是面向业务系统的最终输出。它只表达评估判断及其依据，不包含固定的系统字段代码、字典值或保存结构。

应用层应使用组织评估输入时保留的内部字段绑定，将本结构转换为实际业务字段。转换阶段只处理字段代码、字典保存值、数据类型、记录关系和必填校验，不得重新作出专业判断。

```text
1. 评估结果：通过或未通过
2. 标准化隐患描述
3. 核心事实及证据一致性
4. 成立路径、安全基准和隐患事实
5. 隐患程度和当前状态
6. 隐患分类语义和隐患类型层级
7. 隐患因素及其主导项依据
8. 可能后果、人员伤害严重程度及分析依据
9. 隐患等级、重大隐患匹配结果和判级信息状态
10. 人工补充项
11. 处置要求和整改措施
12. 是否需要提交整改计划
13. 整改时限要求
14. 验收条件
15. 业务必填语义响应
16. 引用依据和判断限制
```

“业务必填语义响应”逐项对应输入的评估字段约束，输出语义标识、字段含义、可读结果、响应状态以及判断依据或缺失信息。它用于保证数据转换阶段不再重新评估。

评估未通过时至少输出：评估结果、三类隐患事实均不成立的客观依据、不符合隐患定义的判断过程，以及其余评估字段约束的不适用或待补充状态。

隐患事实、可能后果和整改措施必须分栏表达，不得合并为一段混合描述。

## 九、质量检查

完成评估前逐项检查：

- 是否始终围绕当前登记隐患？
- 是否把登记描述作为主要观察记录，而不是要求图片独立证明全部内容？
- 是否将核心观察、可能后果和整改建议拆开？
- 是否给出了评估通过或评估未通过的明确结论？
- 评估未通过是否有客观事实证明不属于隐患，而不是笼统写资料不足？
- 信息缺失、证据冲突、分级不足或整改信息不完整是否全部转为人工补充项？
- 是否把人的不安全行为、物的不安全状态和管理缺陷都纳入了隐患识别范围？
- 每项隐患是否有事实，并满足规范偏离或潜在致害中的至少一条成立路径？
- 可能后果是否采用合理可预见的最严重后果，并与隐患描述分开？
- 可能后果描述与业务要求的后果严重程度选项是否分别输出？
- 重大隐患是否完成适用性、必要要件和排除条件检查？
- 整改措施是否直接针对不安全行为、不安全状态或管理缺陷？
- 当场整改是否仍保留发现时点的评估结论？
- 是否逐项响应全部评估字段约束，并只使用当前允许的可读选项？
- 是否把界面默认值当成评估结论，或者为满足必填而编造字段值？
- 待补充和不适用的必填约束是否明确说明原因及其对保存的影响？
