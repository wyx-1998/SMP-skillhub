# 企业资质材料识别

按“识别材料 → 分类 → 发现实体 → 提取字段 → 查询现有记录 → 输出预览 → 用户确认后保存”执行。

## 一、识别和分类材料

1. 对用户提供的图片、扫描件或文档执行 OCR，保留文件名、页码、原始识别文本和置信不足内容。
2. 根据证件标题、发证机关、正文结构和关键字段判断材料类型，文件名只作为辅助线索。
3. 企业资质材料按以下业务语义分类：
   - 营业执照；
   - 安全生产许可证；
   - 企业资质证书；
   - 法人代表资质证书；
   - 爆破作业单位许可证；
   - 安全员证书；
   - 三位一体认证；
   - 近三年安全绩效证明；
   - 单位负责人安全证；
   - 技术人员资格证；
   - 安全管理人员安全证；
   - 安全管理网络图；
   - 安全生产管理制度；
   - 安全操作规程；
   - 安全生产责任制；
   - 其他企业侧审核文件。
4. 同一文件包含多份独立证件时，按证件边界拆成多项识别结果；同一证件跨页时合并为一项，不按页重复建记录。
5. 不能确定材料类型时，列出候选类型和冲突依据，停止该项的实体映射与保存，不用“其他审核文件”掩盖分类不确定。

## 二、发现目标实体

1. 当前上下文已经提供唯一 `entityCode` 时，先调用 `describeEntity(entityCode)` 验证其与材料类型是否一致；不一致或未提供时，按以下顺序发现：
   - 调用 `discoverModules()`，根据模块描述和关键词选择企业资质所属模块；
   - 调用 `listEntitySummaries(module)` 获取该模块的实体索引；
   - 根据材料类型、实体描述和常见叫法确定候选实体；
   - 候选不唯一时，分别调用 `describeEntity(entityCode)`，根据字段、字典、子实体和引用关系确定唯一实体。
2. 目标材料由通用附件实体承载时，使用 `describeEntity` 返回的字典选项确定材料子类型，不固化子类型字段名或保存值。
3. 实体、材料子类型或承载关系不能唯一确定时停止映射，说明缺少的语义信息，不猜测实体编码。
4. `describeWorkflow` 当前不返回流程定义，不得依赖它判断节点、状态、允许动作或保存条件。

## 三、提取和映射字段

1. 只以 `describeEntity` 返回的字段、类型、必填项、字典、子实体和引用关系作为映射依据。
2. 从 OCR 原文逐字段提取数据，并保留“字段值 → 原文位置或页码”的来源对应关系。
3. 字典字段先匹配可读选项，再保留对应保存值；不得把展示文本直接当作字典保存值。
4. OCR 未识别、原文未提供或存在冲突的字段标记为缺失或待确认，不补造证件号码、日期、机关、人员、企业或其他业务信息。
5. 仅提取目标实体存在的字段。关联字段和父级外键必须来自当前上下文、实体关系或已查询记录，不根据历史字段名推断。
6. 保存附件时，只使用系统已经提供的附件标识和元数据，不编造附件 ID、名称或路径。

## 四、查询现有记录

1. 根据 `describeEntity` 返回的实体角色、子实体和引用关系，确认目标实体是独立记录还是需要关联企业主记录。
2. 目标实体需要父级关联时，从当前页面或上下文取得唯一主记录标识，并使用实体关系给出的关联字段；缺少任一信息时仍可输出识别结果，但不得生成可执行保存计划。
3. 调用 `searchRecords(entityCode, where, pageNum, pageSize)` 查询同一业务范围内的现有记录。专用实体使用已确认的关联条件和证件唯一信息，通用附件实体还需使用当前字典映射出的材料子类型；`where` 不得扩大查询范围。
4. 将现有记录区分为可更新记录、重复记录、空记录、已失效记录和应保留历史记录。没有客观规则时不得自行认定为脏数据。
5. 一份文件识别出多份证件时，为每份证件分别确定目标实体、现有记录和新增或更新方式，形成一对多保存计划。

## 五、输出预览

基于识别结果输出可读性高的预览。按材料逐项展示：

- 来源文件和页码；
- 识别的材料类型；
- 目标实体的可读名称；
- 已提取字段及可读值；
- 缺失、冲突和待确认字段；
- 现有记录匹配结果；
- 计划新增、更新、删除或保留的记录；
- 不能保存时的具体阻断原因。

只展示可读值，内部保存计划保留实体字段值和字典保存值。没有明确实体、关联记录或完整必填数据时，不把预览描述成可直接保存。

## 六、确认和执行

1. 先输出完整预览，等待用户明确确认；确认前不调用 `saveRecords`、`deleteRecords` 或 `performAction`。
2. 用户确认后重新调用 `describeEntity`，并用相同唯一条件重新调用 `searchRecords`，确认实体定义、关系和现有记录没有发生影响保存计划的变化。
3. 更新记录时调用 `saveRecords(entityCode, data)` 并携带现有记录 ID；新增记录时不伪造 ID。传入数据只能包含实体允许的字段和已确认的保存值。
4. 只有预览中已经列明、用户已经确认且能够精确定位的重复、空白或错误记录，才调用 `deleteRecords(entityCode, id)`；不得批量扩大删除范围，也不得删除需要保留的历史记录。
5. 多份证件按目标实体分别保存，确保一份独立证件对应一条业务记录，不生成空记录或重复记录。
6. 只有用户明确要求执行独立业务动作，且当前上下文提供准确 `actionName` 时，才调用 `performAction(entityCode, id, actionName, args)`；不得自行推断动作名或合法性。
7. MCP 返回结构化错误时，根据 `type`、`code`、`retryable` 和 `details` 处理。参数或约束错误修正后重试；认证、权限、动作不允许、记录不存在或不可重试错误直接说明，不伪装成执行成功。
8. OCR 结果、材料分类、实体定义、关联对象、现有记录或字段映射发生变化时停止执行，重新输出预览并取得用户确认。

## 执行门禁

- 不使用静态实体编码、历史字段名或经验值替代 MCP 返回的当前实体定义。
- 不绕过 MCP 工具直接拼接业务 HTTP 请求。
- 不把 OCR 推测内容写成已确认字段值。
- 目标实体需要父级关联时，不在缺少主记录或关联字段的情况下保存；任何实体均不得在缺少必填数据、附件标识或必要字典映射时保存。
- 不以“清理脏数据”为由删除未经确认或无法精确定位的记录。
- 没有用户确认，不新增、更新、删除或执行独立业务动作。
